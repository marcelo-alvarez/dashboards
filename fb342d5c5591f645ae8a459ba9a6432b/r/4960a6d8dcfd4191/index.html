Report #4 — 100-realization smoke test: SLURM jobs, I/O, and a walltime finding

roman-covariance · perlmutter-marcelos-claude-code · production sbatch path at scale

TL;DR

  • Purpose: a 100-realization smoke test to validate the production SLURM path at scale before the

full 1883-realization run — right flags, right output location, lean output, and the Hα LFs.

  • The sbatch path works and the lean output is correct at scale. Boxes populate on 8 GPUs and write

lean catalogs (no SFH/MAH histories) to the right place; galaxy counts are full and reproducible.

  • The finding: the 10-minute walltime is too tight under concurrent I/O. Run alone, a box finishes in

~3:56; with dozens writing multi-GB files to CFS at once, boxes take ~10 min and **41 of 68 jobs hit the

walltime (TIMEOUT)**. The science write still completed in every case checked — the catalogs are

complete, not truncated — but the config is fragile and needs a longer walltime.

  • I cancelled the remaining 1133 jobs to stop running a misconfigured walltime; the data already on

disk (65 valid lean catalogs, 6 realizations) is usable.

  • Throughput is GPU-contended: an initial burst of ~58 boxes ran, then the rest queued at "Priority"

behind other users. At-scale wall time is scheduler-dependent.

The walltime finding in detail. The canary (a single box, no contention) ran in 3:56 total (2:34 in the

srun step). Under the smoke-test load — up to ~58 boxes each streaming a 1.4–5.0 GB collective HDF5 write to

CFS simultaneously — per-box elapsed climbed to ~9:54–10:34, against a 10:00 limit, so most jobs were killed

as TIMEOUT. Crucially, galaxy counts in the resulting files are tightly clustered with no low outliers

(z=0.2: ~50.0–50.2M; z=1.1: ~39.1M; z=2.0: ~25.4M), so the writes had finished — the kill lands in MPI

teardown, after the data is on disk. The MPIIO WARNING: DVS stripe width of 48 requested but DVS set it to 1

in every job's stderr points at suboptimal striping as the write bottleneck under load.

Caveat worth recording: the production manager marks a job COMPLETED if its output path exists, which a
TIMEOUT job satisfies — so the tracker would over-count completions. Completion should be gated on a clean
exit / a validated catalog, not path existence.

Lean output at scale

  • Per-box lean size is redshift-dependent: 5.0 GB at z=0.2 → 1.4 GB at z=3.0 (~3.5 GB average), vs ~37 GB

unlean — the ~10× reduction holds at scale. 65 catalogs so far total 233 GB.

  • Full-run extrapolation: 1883 × 12 × ~3.5 GB ≈ ~80 TB on CFS (consistent with the ~85 TB plan).

Bonus — the catalogs are scientifically usable

Because the catalogs are complete, the ensemble Hα LF is already meaningful. This is the **current

(uncalibrated, Kennicutt-placeholder) l_halpha**, accumulated over the 5–6 realizations down per in-band

redshift, against the Nagaraj+2023 observed Hα+[NII] target — i.e. the "before" picture the ensemble

abundance-matching calibration will remap onto the target.

Two things to fix before the full 1883-realization run

1. Walltime + I/O. Raise the per-box walltime to a comfortable margin (≈25–30 min) so the write+teardown

never races the limit, and address the write bottleneck (HDF5/Lustre striping; consider staggering or

capping concurrent writers). This is the direct cause of the TIMEOUT rate.

2. Job packing. 1883 × 12 ≈ 22.6k boxes cannot be 22.6k individual jobs. The manager's array path

(batch_size>1) currently runs bare python with no srun -n 8, so only batch_size: 1 is correct

today — that needs fixing (or a multiple-boxes-per-job driver) before the full run.
