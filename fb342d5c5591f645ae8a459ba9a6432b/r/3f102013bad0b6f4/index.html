The footprint problem (and the fix)

Each populated realization is ~423 GB across 12 redshifts; the full 1883-realization

set in the current format would be ~787 TB. The bulk is two per-galaxy tables —

sfh_table and log_mah_table, each (N_galaxies × 50) float64 (~15 GB *each* at z=1.1) — plus

mah_params/host_mah_params.

These histories are provably unnecessary for the deliverable. Direct check on the z=1.1 mock:

sfh_table[:, t_obs_bin] equals 10logssfr_t_obs · 10logsm_t_obs to machine precision
(Δlog₁₀ = 0.0000 ± 0.0000), and l_halpha = 1.266e41 × SFR exactly (linear KTC94).

So l_halpha, the abundance-matching calibration, and the Roman selection can all be reconstructed

from the instantaneous scalars alone. Two leaner output contracts:

  • Lean catalog — keep pos, vel, l_halpha, l_oii, logsm_t_obs, logssfr_t_obs, logmp_t_obs, upid

+ the small halos/*. ~3.9 GB at z=1.1, ~45 GB/realization,

~85 TB for the full set (~17× smaller). Keeps a reusable galaxy catalog.

  • On-the-fly (your "more on-the-fly spirit") — per realization: populate → calibrate → apply Roman

selection → measure the statistic → write **only the compact data vector + the small Roman-selected

subcatalog** (pos + L_Hα; ~10⁴–10⁵ galaxies, not 39M), then discard the full catalog.

~tens of GB total. Storage stops being a constraint; the GPU populate step becomes the only real cost.

Where the change lives: one localized edit in src/covariance_mocks/hdf5_writer.py

(canonical_specs is derived from the galcat keys at write time) — allowlist the galaxy keys to persist,

and pop the history tables right after add_emission_lines (which also frees memory during the run).
